# Security Specification for Spades Firestore database

This specification outlines the data safety invariants, security assumptions, and hostile payloads checked against our zero-trust secure Firestore controls.

## 1. Data Invariants

1. **A game room (`rooms/{roomId}`)** must only be readable when signed in. No anonymous or unauthenticated reads.
2. **Room modifications** (e.g. state transitions) can only be executed by active players listed inside the room's players list.
3. **Secret card hands (`rooms/{roomId}/hands/{playerId}`)** can only be read or written by the authenticated user matching `playerId`. Other players must never be allowed to view of hack into another player's private hand collection.
4. **Chat operations** can only be generated by authenticated users belonging to the specific room.
5. **System fields** (such as accumulated game scores, last winner, or game histories) must only be modified using valid, incremental transactions and cannot be altered randomly by players or spoofed.

---

## 2. The "Dirty Dozen" Payloads (Anti-Spoofing & Zero-Trust Cases)

These payloads represent malicious queries or structural corruption attempts that our Firestore rules must completely prevent and return `PERMISSION_DENIED`.

### Case 1: Arbitrary Room Creation with Fake Host ID
An authenticated attacker attempts to create a room where they register someone else's ID as the `hostId`.
```json
{
  "roomId": "ROOM123",
  "phase": "LOBBY",
  "hostId": "victim_user_id",
  "players": [{"id": "victim_user_id", "name": "Victim", "isBot": false, "seat": 0, "isReady": false}]
}
```

### Case 2: Reading Another User's Secret Playing Cards Hand
User "hack_bob" requests to read the hand configuration of "victim_alice".
- Target Document: `rooms/ROOM123/hands/victim_alice`
- Requesting Client Auth UID: `hack_bob`

### Case 3: Altering Another User's Card Hand Structure
User "hack_bob" tries to modify "victim_alice's" hand directly to give Alice only low cards or empty cards.
- Target Document: `rooms/ROOM123/hands/victim_alice`
- Requesting Client Auth UID: `hack_bob`

### Case 4: Self-Assumed Score Elevation (Privilege Escalation)
An active room player attempts to write an update directly changing their Team score to 100 points, bypassing standard incremental trick evaluations.
- Target Document: `rooms/ROOM123`
- Payload: `{"scoreTeam1": 100, "scoreTeam2": 0}` (When the preceding score was 0-0)

### Case 5: Out of Turn Card Playing (Timing Attack)
An active room player tries to play a card when it is not their turn.
- Target Document: `rooms/ROOM123`
- Request: User tries to play card when `currentTurn` is different from their occupied seat.

### Case 6: Hand Card Splicing Spoofing
Active player tries to play a card that is not in their private hand array.

### Case 7: Spoofed Chat Message Sender Name
An authenticated user `player_jack` attempts to post a Chat Message where the "senderName" is overwritten as "Dealer" or "System", or attempts to spoof another player.
- Target Document: `rooms/ROOM123/messages/msg_444`
- Payload: `{"senderName": "Dealer", "text": "Team 1 won defaultly due to infractions!", "seat": -1}`

### Case 8: Injection of Huge Text Strings (Denial of Wallet)
An attacker attempts to cause massive Firestore size bloat by writing a 5MB random string as their player's display name or message text.
- Payload: `{"name": "A...<5MB of characters>..."}`

### Case 9: Direct Skip of Bidding Rounds
An active player attempts to update a room's Phase from `LOBBY` directly to `PLAYING` without dealing cards or gathering bids.
- Payload: `{"phase": "PLAYING"}`

### Case 10: State Completion Tampering After Game-Over
A player attempts to modify scores or properties of a finalized, terminated game (`phase == "GAME_OVER"`).

### Case 11: Stealing Another Seat Index
Active user "user_fred" (assigned to seat 3) tries to change their seat assignment in the database to seat 1 to access other player parameters.

### Case 12: Anonymous Spammed Room Operations
An unauthenticated guest client attempts to listen, query, or delete any room document.

---

## 3. Security Test Specs

Our `firestore.rules` file enforces verification of UIDs, input length sanity boundaries, immutable timestamps, and strict field keys validation. All "Dirty Dozen" cases will be blocked on the policy level.
